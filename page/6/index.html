手写一个自带kv-cache的attention,然后和pytorch的build-in attention进行性能和精度对比
几个关键的问题：

  1. 怎么计算有KV-Cache后，计算复杂度/FLOPS等方面优化了多少？
    对于prefill时
  2. 给定一个模型的超参数，怎么确定KVCache需要多少Blocks?

Objectives

  1. 手搓mha: 精度和pytorch build-in的multi-head-attention一致

Questions

  1. 还得是通过手搓公式的方法来进行理解：

假设我们有一个长度为3的seq,做完prefill后要预测next token
通过embedding后，$ Q = [q0, q1, q2]^T $ , $ K = [k0,k1,k2] $
$$ S = QK = [qi^Tkj] $$
通过attention计算和掩码处理后，假设我们得到的attentore score =
$$
\begin{bmatrix}
1 & 0 & 0 \
0.2 & 0.8 & 0 \
0.1 & 0.2 & 0.7 \
\end{bmatrix}
$$
那么atten和V矩阵相乘，得到
$$
\begin{bmatrix}
v0 \
0.2v0 + 0.8v1 \
0.1v0 + 0.2v1 + 0.7v2\
\end{bmatrix}
$$
且对于next token的预测，使用的输入是最后一行的0.1v0 + 0.2v1 + 0.7v2这个结果作为线性变换层的输入

2. KV Cache开启前后的FLOPs差异

假设序列长度为L，模型维度d，头数h，头维度d_h = d/h，batch size为B：

  • 无KV Cache：

    • 每次计算都需要处理整个序列：O(B*L^2*d)

    • 自回归生成N个token的总FLOPs：O(B*N*L^2*d)（随着L增长）

  • 有KV Cache：

    • Prefill阶段：O(B*L^2*d)（与无Cache相同）

    • 解码阶段：每次只计算新token的Q与所有K的点积：O(B*L*d)（每个token）

    • 自回归生成N个token的总FLOPs：O(B*L^2*d + B*N*L*d)

  1. 对于一个batch 是不是做完prefill之后，P节点的kv-cache就用不到了？-> 分析一下当前LLM是怎么管理KVCache的释放操作的

3. KV Cache的内存管理

内存计算：

  • 每个token的KV Cache大小：2 * B * h * L * d_h = 2 * B * L * d（因为h * d_h = d）

  • 对于N个token，Cache大小：2 * B * N * d（假设L = N）

管理策略：

  1. 静态分配：

    • 预分配固定大小的Cache（如最大长度限制）。

    • 简单但可能浪费内存或限制生成长度。

  2. 动态分配：

    • 按需扩展Cache内存（如使用torch.cat）。

    • 更灵活但可能引入内存碎片。

  3. Block管理（如PagedAttention）：

    • 将Cache划分为固定大小的Block（如每Block存储16个token的KV）。

    • 维护一个逻辑到物理Block的映射表。

    • 支持不连续的物理内存，便于共享和释放。

释放策略：

  • Prefill阶段：通常不需要释放，因为所有token都用于生成第一个输出。

  • 解码阶段：

    • 若支持可变输入长度（如多个并行序列），可以按序列释放。

    • 使用引用计数或标记-清除策略管理Block。

4. Swap-in/Swap-out策略

当GPU内存不足时：

  1. 最近最少使用（LRU）：

    • 将最久未访问的Cache Block交换到CPU内存。

    • 需要时再交换回GPU。

  2. 按重要性交换：

    • 根据注意力权重或梯度信息决定保留哪些Block。

    • 高权重的Block保留在GPU。

  3. 预取（Prefetch）：

    • 预测下一步需要的Block提前加载。

    • 例如，基于当前生成的token预测下一个可能访问的Block。

编译原理与优化技术总结

一、编译原理基础概念

1. 基本块 (Basic Block)

  • 定义：一个顺序执行的指令序列，只有一个入口点（第一条指令）和一个出口点（最后一条指令）
  • 特点：
    • 只能从第一条指令进入
    • 只能从最后一条指令退出
    • 内部没有跳转指令
  • 划分方法：
    • 第一条指令作为基本块的入口
    • 跳转指令的目标地址作为基本块的入口
    • 跳转指令后的指令作为新基本块的入口

2. 控制流图 (Control Flow Graph, CFG)

  • 定义：表示程序控制流的有向图
  • 组成：
    • 节点：基本块
    • 边：基本块之间的控制转移
  • 构建方法：
    • 每个基本块作为图中的一个节点
    • 如果基本块A可能跳转到基本块B，则添加从A到B的边
  • 用途：
    • 程序分析
    • 优化
    • 代码生成

3. 数据流图 (Data Flow Graph)

  • 定义：表示程序中数据依赖关系的有向图
  • 组成：
    • 节点：操作或变量
    • 边：数据依赖关系
  • 类型：
    • 定义-使用链
    • 使用-定义链
  • 用途：
    • 变量分析
    • 优化
    • 并行化

二、编译器优化技术

1. 局部优化

1.1 常量折叠 (Constant Folding)

  • 原理：
    • 在编译时计算常量表达式
    • 将常量表达式替换为其计算结果
  • 实现方法：

1.2 常量传播 (Constant Propagation)

  • 原理：
    • 将变量的常量值传播到使用点
    • 替换变量引用为常量值
  • 实现方法：

1.3 代数简化 (Algebraic Simplification)

  • 原理：
    • 应用代数规则简化表达式
    • 消除冗余计算
  • 示例：
    • x + 0 → x
    • x * 1 → x
    • x * 0 → 0

2. 全局优化

2.1 全局值编号 (Global Value Numbering)

  • 原理：
    • 为每个计算分配唯一编号
    • 识别并重用相同计算
  • 实现步骤：
    1. 构建值依赖图
    2. 为每个计算分配编号
    3. 识别相同计算
    4. 替换重复计算

2.2 全局公共子表达式消除 (Global Common Subexpression Elimination)

  • 原理：
    • 识别跨基本块的重复计算
    • 重用计算结果
  • 实现方法：

3. 循环优化

3.1 循环展开 (Loop Unrolling)

  • 原理：
    • 将循环体复制多次
    • 减少循环次数
  • 实现方法：

3.2 循环不变代码外提 (Loop Invariant Code Motion)

  • 原理：
    • 将循环中不随迭代变化的计算移到循环外
  • 实现步骤：
    1. 识别循环不变表达式
    2. 检查移动的安全性
    3. 将代码移到循环外

3.3 循环向量化 (Loop Vectorization)

  • 原理：
    • 将循环中的标量运算转换为向量运算
    • 利用SIMD指令提高性能
  • 实现条件：
    • 循环迭代间无依赖
    • 数据对齐
    • 目标架构支持向量运算

4. 函数级优化

4.1 函数内联 (Function Inlining)

  • 原理：
    • 将函数调用替换为函数体
    • 消除函数调用开销
  • 实现方法：

4.2 尾递归优化 (Tail Recursion Optimization)

  • 原理：
    • 将尾递归转换为循环
    • 消除递归调用开销
  • 实现条件：
    • 递归调用是函数的最后操作
    • 递归调用的返回值直接返回

5. 指令级优化

5.1 指令调度 (Instruction Scheduling)

  • 原理：
    • 重排指令顺序
    • 充分利用CPU流水线
  • 实现方法：
    • 构建依赖图
    • 使用启发式算法重排指令

5.2 寄存器分配 (Register Allocation)

  • 原理：
    • 将变量映射到寄存器
    • 最小化内存访问
  • 实现算法：
    • 图着色算法
    • 线性扫描算法

三、数据流分析

1. 活跃变量分析 (Live Variable Analysis)

  • 定义：确定变量在程序点是否可能被使用
  • 用途：
    • 死代码消除
    • 寄存器分配
  • 实现方法：

2. 到达定义分析 (Reaching Definition Analysis)

  • 定义：确定定义点是否可能到达使用点
  • 用途：
    • 常量传播
    • 死代码消除
  • 实现方法：

3. 可用表达式分析 (Available Expression Analysis)

  • 定义：确定表达式在程序点是否可用
  • 用途：
    • 公共子表达式消除
    • 代码移动
  • 实现方法：

四、优化效果评估

1. 性能指标

  • 执行时间
  • 代码大小
  • 内存使用
  • 编译时间

2. 优化策略选择

  • 根据目标平台特性
  • 考虑编译时间开销
  • 权衡优化效果

3. 调试和验证

  • 正确性验证
  • 性能测试
  • 回归测试

这个总结涵盖了编译原理和优化技术的主要概念，可以帮助你准备面试。建议你重点掌握：

  1. 基本概念的定义和关系
  2. 各种优化技术的原理和实现方法

  4. 优化效果评估的方法
